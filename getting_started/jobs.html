    Jobs¬∂

    Given a running MiniCluster that exposes a RESTful API, you have several modes for interaction:

    1. Submit jobs in the user interface (basic)

    2. Submit jobs from the command line ‚ÄúRESTful‚Äù or within Python (custom)

    3. Use a helper tool ‚Äúflux-cloud‚Äù

    4. Submit jobs interacting with Flux via ssh-ing to the pod (advanced)

    We provide clients to interact with a running server, and each will be briefly shown below. The Python client is also included in our Tutorials.

    Credentials¬∂

    Whether you use a command line client, a language SDK, or the web interface - you need your credentials! The credentials are printed in the log of the index-0. These are the FLUX_USER and FLUX_TOKEN environment variables. A client requires exported them to the environment, and the web interface requires copy paste into a form.

    # get the job pod identifier for index 0, the broker
$ kubectl get -n flux-operator pods
$ kubectl logs -n flux-operator ${brokerPod}

    üîë Your Credentials! These will allow you to control your MiniCluster with flux-framework/flux-restful-api
export FLUX_TOKEN=9c605129-3ddc-41e2-9a23-38f59fb8f8e0
export FLUX_USER=flux

üåÄsudo -u flux flux start -o --config /etc/flux/config ...

    Note that they appear slightly above the command, which isn‚Äôt at the bottom of the log!

    Ways to Submit¬∂

    1. User Interface¬∂

    To see the user interface, you‚Äôll first need to do some kind of port forwarding:

    kubectl port-forward -n flux-operator flux-sample-0-zdhkp 5000:5000
Forwarding from 127.0.0.1:5000 -> 5000

    This should be exposed at http://127.0.0.1:5000. It‚Äôs not terribly informative, so we won‚Äôt show the main portal here! ü§≠

    After your MiniCluster is created with a RESTFul API, the easiest thing to do is open the user interface to http://127.0.0.1:5000 After port forwarding in a terminal and then submitting in the job UI (you will need to login with the flux user and token presented in the terminal). Here is the example command to try for the basic tutorial here:

    lmp -v x 2 -v y 2 -v z 2 -in in.reaxc.hns -nocite

    And the workdir for this default example needs to be:

    And you can also try using the RESTFul API Clients to submit instead (discussed next). Note that there is currently no way for the RESTful client to ask to destroy the cluster - you‚Äôll still need to use kubectl to do that.

    Or do the same from within Python:

    from flux_restful_client.main import get_client

cli = get_client()

# Define our jop
command = "lmp -v x 2 -v y 2 -v z 2 -in in.reaxc.hns -nocite"

# Submit the job to flux
print(f"üò¥ Submitting job: {command}")
res = cli.submit(command=command)
print(json.dumps(res, indent=4))

# Get job information
print("üçì Getting job info...")
jobid = res['id']
res = cli.jobs(jobid)
print(json.dumps(res, indent=4))

# Get job output log
print("üçï Getting job output...")
res = cli.output(jobid)
print(json.dumps(res, indent=4))

    3. Flux Cloud Helper Tool¬∂

    If you have a set of experiments you want to run on a production cluster, and especially if you don‚Äôt want the cluster being up longer than it has to be, check out Flux Cloud. It‚Äôs a small experiment runner wrapper that makes bringing up the cluster, installing the operator, running a matrix of experiments, and destroying the cluster much easier than all the copy pasting of commands typically required!

    4. Advanced Shell¬∂

    You can also submit jobs interacting with Flux via ssh-ing to the pod! This is considered advanced, and is good for debugging. As you did before, get your pod listing:

    $ kubectl -n flux-operator pods

    And then shell in

    $ kubectl exec --stdin --tty -n flux-operator ${brokerPod} -- /bin/bash

    Note that if you have more than one container, you‚Äôll need to include it‚Äôs name with -c <container>. Also note that if you need to load any custom environments (e.g., something you‚Äôd define in the preCommand for a container) you‚Äôll likely need to do that when you shell in.
